## 1. What Zeus_osVM is

A small, educational **stack-based virtual machine** written in C11. It runs its
own bytecode, has its own assembler/disassembler, and — the whole point — exposes
**networking as first-class opcodes**: TCP/UDP sockets and raw IP packets.

- Operands live on a **64-bit signed integer** stack.
- Programs are written in a tiny assembly language (`.zasm`), assembled to
  bytecode (`.zeus`), then executed.
- One binary, `zeus`, does everything via subcommands.

## 2. Build & install

From the repo root (`/home/dennisjcarroll/Desktop/cc_zone/zeus`):

```bash
make            # builds zeus, zasm, zdis
make test       # runs unit tests (expect 7/7)
make smoke      # full end-to-end check against every example
make clean      # remove binaries, *.o, examples/*.zeus, zasm/zdis wrappers
```

Requirements: a C11 compiler (`gcc` or `clang`) and `make`. Networking uses
plain BSD sockets — no external libraries. Override the compiler with
`make CC=clang`.

Three artifacts are produced:

| File   | What it is                                               |
|--------|----------------------------------------------------------|
| `zeus` | The VM + assembler + disassembler (single binary)        |
| `zasm` | Shell wrapper → `./zeus asm`  (recreated by `make`)       |
| `zdis` | Shell wrapper → `./zeus dis`  (recreated by `make`)       |

> `zasm`/`zdis` are generated by the Makefile and deleted by `make clean`. If
> they go missing, run `make`.

## 3. The `zeus` CLI

`zeus` dispatches on its first argument (`src/main.c`):

```bash
zeus run     <file.zeus>              # execute a bytecode file
zeus asm     <in.zasm> <out.zeus>     # assemble source -> bytecode
zeus dis     <file.zeus>              # disassemble bytecode -> instructions
zeus run-asm <in.zasm>                # assemble in memory and run immediately
```

`zasm in.zasm out.zeus` == `zeus asm in.zasm out.zeus`, and `zdis file.zeus` ==
`zeus dis file.zeus`.

### Typical loops

Fastest edit-run cycle:
```bash
./zeus run-asm examples/hello.zasm
```

Inspect what the assembler produced:
```bash
./zasm examples/hello.zasm hello.zeus
./zdis hello.zeus           # see the actual instructions + offsets
./zeus run hello.zeus
```

Example disassembly (note: jump targets are **absolute code offsets**):

## 4. The smoke test

`make smoke` (or `./smoke_test.sh`) builds, runs the unit tests, and exercises
every example end to end: hello, asm/dis round-trip, boot splash, the TCP echo
server, the HTTP server, and raw ping. It prints a `PASS/FAIL/SKIP` summary and
exits non-zero if anything required fails, so it works as a CI gate.

Configuration via environment variables:

```bash
ECHO_PORT=23456 HTTP_PORT=9090 ./smoke_test.sh   # avoid busy ports
SKIP_NET=1 ./smoke_test.sh                        # skip server tests
SKIP_RAW=1 ./smoke_test.sh                        # skip raw ping
sudo ./smoke_test.sh                              # send a real raw packet
```

The raw-ping check inverts by privilege: **unprivileged**, success means the raw
opcodes degraded to `-1` without crashing (the designed behavior); **as root**,
success means a real packet went out.

## 5. Execution model

### Two separate address spaces (important)

This trips people up. Zeus has **two** distinct "address 0"s:

1. **Code space** — the assembled instruction bytes (`code` buffer). The program
   counter (`pc`) indexes into this. Jump/call targets (labels in `.code`) are
   **byte offsets into code**.
2. **Data/memory space** — a flat **64 KB** byte array (`memory[]`). All
   `LOAD`/`STORE`/`SEND`/`RECV`/string operations use these addresses. At load
   time the `.data` section is copied to **memory address 0**, so a label in
   `.data` is simply its byte offset within the data section.

A label's meaning therefore depends on which section it's in:
- Label in `.code` → an instruction address (use with `JMP`, `JZ`, `JNZ`, `CALL`).
- Label in `.data` → a memory address (use with `PUSH`, then `LOAD*`/`STORE*`/net ops).

Code is **not** addressable as data, and data is **not** executable.

### Stacks and registers

| State        | Purpose                                  | Size                         |
|--------------|------------------------------------------|------------------------------|
| operand stack| 64-bit signed values; all computation    | 8192 slots (`ZEUS_STACK_SIZE`) |
| call stack   | return addresses for `CALL`/`RET`        | 1024 (`ZEUS_CALL_SIZE`)      |
| memory       | flat byte-addressable RAM                | 65536 bytes (`ZEUS_MEMORY_SIZE`) |
| `pc`         | program counter (offset into code)       | —                            |

Execution starts at the header's `entry_point` (always `0` from the assembler)
and runs one instruction at a time until `HALT` or an error. The interpreter
core is `zeus_vm_step`; `zeus_vm_run` just loops it.

### Endianness & word size

Immediates and 64-bit memory cells are **little-endian, signed 64-bit**.

## 6. The assembly language

A `.zasm` file is line-oriented text. Each line may have a label, then either a
**directive** (starts with `.`) or an **instruction**, then an optional comment.

```asm
label:  INSTRUCTION operand   ; comment
```

### Comments
`;` to end of line. Blank lines are ignored.

### Labels
`name:` at the start of a line. Names may contain letters, digits, `_`, and `.`.
A label records the current offset **in whatever section is active**. Labels are
resolved in a second pass, so forward references are fine.

### Sections
- `.code` — instructions (default at start of file).
- `.data` — raw bytes/strings/words loaded into memory at address 0.

You can switch back and forth; each section accumulates independently.

### Directives

| Directive        | Effect                                                              |
|------------------|--------------------------------------------------------------------|
| `.code`          | Switch to the code section                                          |
| `.data`          | Switch to the data section                                          |
| `.byte N`        | Emit one byte (`N & 0xFF`)                                          |
| `.word N`        | Emit an 8-byte little-endian value (also accepts a label)          |
| `.ascii "..."`   | Emit string bytes, **no** trailing NUL                             |
| `.asciz "..."`   | Emit string bytes **plus** a trailing `\0`                         |
| `.org N`         | Pad the current section with zeros up to offset `N` (can't go back) |

`.org` is how you reserve a scratch buffer (see the echo server: `buf: .org 1024`).

### Operands

An instruction operand is either a **number** or a **label**.

- Numbers parse with C rules (`strtoll` base 0): `42`, `-7`, `0x1F`, `010` (octal).
- A label resolves to its offset (code offset or memory address per its section).

### String escapes

Inside `.ascii`/`.asciz`: `\n \t \r \0 \\ \"`. Unknown escapes pass the next
char through literally. Single-string literals are capped at **8192 bytes**.

## 7. Instruction reference

Opcodes are 8-bit (`include/isa.h`). Instructions marked **+imm** are followed
by an 8-byte immediate in the bytecode; everything else takes its inputs from
the stack.

Stack effects are written `( before -- after )` with the **top of stack on the
right**. So `( a b -- a+b )` means `b` was on top.

### Control / halt
| Mnemonic | imm | Stack         | Notes                                            |
|----------|-----|---------------|--------------------------------------------------|
| `HALT`   |     | `( -- )`      | Stop execution                                   |
| `JMP a`  | +imm| `( -- )`      | Unconditional jump to **absolute** code offset `a` |
| `JZ a`   | +imm| `( x -- )`    | Jump to `a` if `x == 0`                           |
| `JNZ a`  | +imm| `( x -- )`    | Jump to `a` if `x != 0`                           |
| `CALL a` | +imm| `( -- )`      | Push return address, jump to `a`                 |
| `RET`    |     | `( -- )`      | Pop return address, jump to it                   |

> The `include/isa.h` comment calls `JMP` "relative", but the implementation
> (`src/vm.c`) sets `pc = a` **absolutely**. Use absolute offsets / labels.

### Stack
| Mnemonic | imm | Stack                | Notes                                     |
|----------|-----|----------------------|-------------------------------------------|
| `PUSH n` | +imm| `( -- n )`           | Push immediate (number or label)          |
| `POP`    |     | `( a -- )`           | Discard top                               |
| `DUP`    |     | `( a -- a a )`       | Duplicate top                             |
| `SWAP`   |     | `( a b -- b a )`     | Swap top two                              |
| `PICK n` | +imm| `( -- x )`           | Copy the `n`-th item from top (`0` = top) |
| `ROLL n` | +imm| —                    | Move the `n`-th item to the top           |

### Arithmetic (signed 64-bit)
| Mnemonic | Stack            | Notes                          |
|----------|------------------|--------------------------------|
| `ADD`    | `( a b -- a+b )` |                                |
| `SUB`    | `( a b -- a-b )` |                                |
| `MUL`    | `( a b -- a*b )` |                                |
| `DIV`    | `( a b -- a/b )` | Error `DIV_ZERO` if `b == 0`   |
| `MOD`    | `( a b -- a%b )` | Error `DIV_ZERO` if `b == 0`   |
| `NEG`    | `( a -- -a )`    |                                |

### Bitwise
| Mnemonic | Stack             |
|----------|-------------------|
| `AND`    | `( a b -- a&b )`  |
| `OR`     | `( a b -- a\|b )` |
| `XOR`    | `( a b -- a^b )`  |
| `NOT`    | `( a -- ~a )`     |
| `SHL`    | `( a b -- a<<b )` |
| `SHR`    | `( a b -- a>>b )` |

### Comparison (push 1 or 0)
`EQ` `NE` `LT` `GT` `LE` `GE` — all `( a b -- (a OP b) )`, e.g. `LT` pushes
`1` if `a < b` else `0`.

### Memory
| Mnemonic   | imm | Stack              | Notes                                            |
|------------|-----|--------------------|--------------------------------------------------|
| `LOAD a`   | +imm| `( -- v )`         | Push 64-bit `memory[a]`                           |
| `STORE a`  | +imm| `( v -- )`         | Store 64-bit `v` at `memory[a]`                   |
| `LOADB a`  | +imm| `( -- b )`         | Push `memory[a]` as 1 byte, **sign-extended**     |
| `STOREB a` | +imm| `( v -- )`         | Store low byte of `v` at `memory[a]`              |
| `LOADI`    |     | `( addr -- v )`    | Pop address, push 64-bit `memory[addr]`           |
| `STOREI`   |     | `( v addr -- )`    | Pop address then value; store 64-bit at `memory[addr]` |
| `LOADBI`   |     | `( addr -- b )`    | Pop address, push byte (sign-extended)            |
| `STOREBI`  |     | `( v addr -- )`    | Pop address then value; store low byte            |

The `*I` ("indirect") variants take the address from the stack; the others bake
it into the instruction as an immediate.

### I/O
| Mnemonic | Stack          | Notes                                            |
|----------|----------------|--------------------------------------------------|
| `PRINT`  | `( a -- )`     | Print `a` as a decimal integer + newline          |
| `PRINTC` | `( a -- )`     | Print low byte of `a` as an ASCII character       |
| `READ`   | `( -- c )`     | Read one char from stdin (`-1` on EOF), push it   |

### Sockets
All return values: `0`/`fd`/byte-count on success, `-1` on failure.

| Mnemonic   | Stack                       | Notes                                         |
|------------|-----------------------------|-----------------------------------------------|
| `SOCK_TCP` | `( -- fd )`                 | Create TCP socket (or `-1`)                    |
| `SOCK_UDP` | `( -- fd )`                 | Create UDP socket (or `-1`)                    |
| `BIND`     | `( fd port -- result )`     | Bind socket to `port`                          |
| `LISTEN`   | `( fd backlog -- result )`  | Mark socket listening                          |
| `ACCEPT`   | `( fd -- new_fd )`          | Accept a connection                            |
| `CONNECT`  | `( fd addr port -- result )`| `addr` = memory address of a host string (e.g. `"127.0.0.1"`) |
| `SEND`     | `( fd buf len -- sent )`    | Send `len` bytes from `memory[buf]`            |
| `RECV`     | `( fd buf len -- received )`| Receive up to `len` bytes into `memory[buf]`   |
| `CLOSE`    | `( fd -- result )`          | Close socket                                   |

### Raw packets
Require `CAP_NET_RAW`/root; otherwise return `-1` (no crash). Max 16 live
packets, max 65535 bytes each.

| Mnemonic          | Stack                          | Notes                                  |
|-------------------|--------------------------------|----------------------------------------|
| `RAW_ALLOC`       | `( size -- pkt_id )`           | Allocate a packet buffer (`-1` on fail)|
| `RAW_FREE`        | `( pkt_id -- result )`         | Free a packet                          |
| `RAW_SET_PROTO`   | `( pkt_id proto -- result )`   | Set IP protocol number                 |
| `RAW_SET_DST`     | `( pkt_id ip -- result )`      | Set destination IP (as integer)        |
| `RAW_SET_SRC`     | `( pkt_id ip -- result )`      | Set source IP (as integer)             |
| `RAW_SET_PAYLOAD` | `( pkt_id buf len -- result )` | Copy payload from `memory[buf]`        |
| `RAW_SEND`        | `( pkt_id iface -- result )`   | Send; `iface` = memory addr of name, or `0` for default |
| `RAW_RECV`        | `( pkt_id buf len -- received )`| Receive into `memory[buf]`            |

## 8. Worked examples

### 8.1 Arithmetic — `(2 + 3) * 4`, print 20
```asm
.code
start:
    PUSH 2
    PUSH 3
    ADD          ; -> 5
    PUSH 4
    MUL          ; -> 20
    PRINT        ; prints "20"
    HALT
```
```bash
./zeus run-asm math.zasm
```

### 8.2 Print a string (how `hello` works)
There is no "print string" opcode — you walk the bytes until the NUL terminator.
```asm
.data
msg:    .asciz "Hi!\n"

.code
start:
    PUSH msg        ; address of first byte
loop:
    DUP             ; keep a copy of the address
    LOADBI          ; load byte at that address
    DUP             ; copy the byte
    JZ done         ; byte == 0 -> end of string
    PRINTC          ; print the char
    PUSH 1
    ADD             ; address += 1
    JMP loop
done:
    POP             ; drop the 0 byte
    POP             ; drop the address
    HALT
```

### 8.3 Subroutine with `CALL`/`RET`
```asm
.code
start:
    PUSH 10
    PUSH 32
    CALL add_two     ; result (42) left on stack
    PRINT            ; "42"
    HALT

add_two:             ; ( a b -- a+b )
    ADD
    RET
```
`CALL` pushes the return address onto the call stack; `RET` pops it. Arguments
and results travel on the operand stack.

### 8.4 Memory scratch with `STOREI`/`LOADI`
```asm
.data
slot:   .word 0       ; 8-byte cell at memory address 0

.code
start:
    PUSH 99
    PUSH slot
    STOREI            ; memory[slot] = 99   ( value addr -- )
    PUSH slot
    LOADI             ; push memory[slot]   ( addr -- value )
    PRINT             ; "99"
    HALT
```

## 9. Networking guide

### 9.1 TCP echo server (annotated)

`examples/echo_server.zasm` (port 12345). The flow:

1. `SOCK_TCP` → fd
2. `BIND` fd to the port
3. `LISTEN`
4. `ACCEPT` → client fd, stash it in memory
5. loop: `RECV` into a buffer; if 0 bytes, done; else `SEND` them back
6. `CLOSE`

Run and test it:
```bash
./zeus run-asm examples/echo_server.zasm &     # background
sleep 1
printf 'hello zeus\n' | nc -w2 127.0.0.1 12345 # -> hello zeus
kill %1
```

Note the recurring pattern `PUSH <label>; LOADI` to read an 8-byte variable out
of the data section, and `PUSH buf; PUSH 1024` to hand a buffer + length to
`RECV`/`SEND`.

### 9.2 HTTP server
```bash
./zeus run-asm examples/http_server.zasm &      # port 8080
sleep 1
curl -s http://127.0.0.1:8080/
kill %1
```

### 9.3 A TCP client (sketch)
```asm
.data
host:   .asciz "127.0.0.1"
msg:    .asciz "ping\n"

.code
start:
    SOCK_TCP            ; ( -- fd )
    DUP                 ; keep fd
    PUSH host           ; address of host string
    PUSH 12345          ; port
    CONNECT             ; ( fd addr port -- result )
    POP                 ; ignore connect result for brevity
    ; fd still on stack
    PUSH msg
    PUSH 5
    SEND                ; ( fd buf len -- sent )
    POP
    CLOSE
    POP
    HALT
```

### 9.4 Raw packets
```bash
sudo ./zeus run-asm examples/raw_ping.zasm
```
Without privileges the raw opcodes return `-1` and the program continues — by
design. Check results on the stack rather than assuming success.

## 10. Bytecode file format

A `.zeus` file is a 17-byte header followed by the code bytes, then the data
bytes (`include/bytecode.h`):

| Field         | Size | Notes                                  |
|---------------|------|----------------------------------------|
| `magic`       | 4    | ASCII `"ZEUS"`                          |
| `version`     | 1    | currently `1` (`ZEUS_VERSION`)          |
| `entry_point` | 4    | start offset into code (assembler: `0`) |
| `code_len`    | 4    | length of code section                  |
| `data_len`    | 4    | length of data section                  |
| code…         | var  | instruction bytes                       |
| data…         | var  | data bytes (copied to memory addr 0)    |

The loader rejects a file whose magic or version doesn't match. **If you ever
change the on-disk layout, bump `ZEUS_VERSION` in `include/zeus.h`.**

## 11. Errors & debugging

The VM stops with one of these (`include/zeus.h`): stack under/overflow, call
stack under/overflow, out-of-bounds (bad code or memory address), divide by
zero, invalid opcode, the `NET_*` family, packet errors, and I/O errors.

Debugging workflow:

1. **Disassemble** to confirm the assembler emitted what you expect:
   ```bash
   ./zasm prog.zasm prog.zeus && ./zdis prog.zeus
   ```
   Check that jump targets point where you think (they're absolute offsets).
2. **Watch the stack discipline.** Most bugs are a missing `POP`/`DUP` or wrong
   operand order. Re-read the `( before -- after )` effects in §7.
3. **Confirm which address space a label is in.** A `.code` label used where a
   memory address is expected (or vice-versa) silently points at the wrong byte.
4. **Networking returns `-1` instead of throwing.** Test the return value;
   don't assume a socket call succeeded.

> A single-step engine (`zeus_vm_step`) exists in `src/vm.c` and powers the run
> loop, but there is no interactive step/debugger CLI yet. Wiring one up is a
> natural next feature — the core already supports it.

## 12. Gotchas

- **Jumps are absolute**, not relative — despite the `isa.h` comment on `JMP`.
- **Two address spaces.** Code offsets (for `JMP`/`CALL`) and memory addresses
  (for `LOAD`/`STORE`/net) both start at 0 and are unrelated.
- **No print-string opcode** — loop over bytes until the `\0` (use `.asciz`).
- **`STOREI`/`STOREBI` pop address first, then value** — push value, then address.
- **String literals max 8192 bytes** (`src/asm.c`).
- **Networking is single-threaded and blocking** — a server blocks in `ACCEPT`;
  run it with `&` and `kill` it, or use two terminals. No `select`/`poll` yet.
- **Raw packets need `CAP_NET_RAW`/root**; otherwise they degrade to `-1`.
- **`make clean` deletes `examples/*.zeus`** and the `zasm`/`zdis` wrappers —
  rerun `make` to get them back.
